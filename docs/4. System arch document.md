- Architectural style
- System components
- Layer responsibilities
- Technology stack
- Deployment topology
- Design decisions and rationale

This document serves as the technical blueprint of the system.

## 2. Architectural Overview

### 2.1 Architectural Style

The system follows a:

- Monolithic Architecture
- Layered (Clean Separation of Concerns) Design
- RESTful API Pattern
- Stateless Authentication Model (JWT)

The architecture is intentionally simplified for small-scale deployment while maintaining professional standards.

## 5. Layered Architecture

The backend is structured into strict layers:

```text
Controller Layer
       ↓
 Service Layer
       ↓
 Repository Layer
       ↓
    Database
```

---

### 5.1 Controller Layer

Responsibilities:

- Handle HTTP requests and responses
- Validate input DTOs
- Enforce role-based access
- Delegate business logic to services

Controllers do NOT:

- Contain business logic
- Access repositories directly

---

### 5.2 Service Layer

Responsibilities:

- Implement business rules
- Enforce ownership validation
- Manage transactions (Transaction boundary defined at service layer using `@Transactional`)
- Coordinate repository calls
- Handle status transitions

All critical logic resides here.

---

### 5.3 Repository Layer

Responsibilities:

- Database access via JPA
- CRUD operations
- Pagination queries
- Custom query methods

Repositories do NOT:

- Contain business rules

---

### 5.4 Security Layer

Spring Security Filter Chain handles:

- JWT validation
- Authentication extraction
- Role authorization

Security enforcement occurs at:

- Filter level (authentication)
- Controller level (role restrictions)
- Service level (ownership checks)

---

## 6. Authentication Architecture

### 6.1 JWT Flow

1. User logs in.
2. Backend validates credentials.
3. JWT token generated with:
   - user_id
   - role
   - expiration
4. Client stores token.
5. Token sent in Authorization header for protected endpoints.

---

### 6.2 Stateless Design

The system is stateless:

- No server-side session storage.
- Each request is authenticated independently.
- Enables horizontal scalability if needed.

## 13. Design Decisions & Rationale

| Decision | Rationale |
| -------- | --------- |
| Monolithic architecture | Simpler for small-scale usage |
| JWT stateless auth | Enables scalability and simplicity |
| JPA ORM | Rapid development and maintainability |
| UUID primary keys | Safer public exposure |
| No Redis | Unnecessary for small scale |
| No microservices | Avoid overengineering |

## 15. Summary

The system architecture:

- Is a clean, layered monolithic design.

- Small-scale deployment
- Learning and demonstration

# Non-Functional Requirements (NFR)

**Intern Job Tracking Application**  
Version: 1.0  
Status: Final  

---

## 1. Introduction

### 1.1 Purpose

This document defines the non-functional requirements (NFRs) of the Intern Job Tracking Application. These requirements specify system quality attributes such as performance, security, scalability, reliability, and maintainability.

This document focuses on **how well** the system must perform its functional requirements.

---

## 2. Performance Requirements

### 2.1 Response Time

| Operation | Maximum Acceptable Response Time |
| --------- | -------------------------------- |
| Login | ≤ 500 ms |
| Internship Listing (paginated) | ≤ 700 ms |
| Apply to Internship | ≤ 800 ms |
| Update Application Status | ≤ 800 ms |

*Measured under normal load conditions (<100 concurrent users) on a warm instance.*

---

### 2.2 Throughput

- The system shall support up to 100 concurrent active users.
- The system shall handle up to 10 application submissions per second without failure.
- The database shall support up to 5,000 total stored applications without performance degradation.

---

### 2.3 Resource Utilization

- Application memory usage shall not exceed 512 MB under standard load.
- Database connection pool shall be configured appropriately (e.g., 10–20 connections).
- CPU utilization should remain under 70% during peak usage.

---

## 3. Scalability Requirements

Although designed for small-scale usage, the system shall:

- Support horizontal scaling of backend if deployed on scalable infrastructure.
- Use stateless JWT authentication to allow multiple backend instances.
- Avoid file storage within backend server.

Explicitly Not Required:

- Distributed microservices
- Event streaming systems
- Database partitioning

---

## 4. Security Requirements

### 4.1 Authentication Security

- Passwords must be hashed using BCrypt.
- JWT must be signed using a secure HMAC algorithm.
- JWT must expire (e.g., 24-hour expiration maximum).

---

### 4.2 Authorization Security

- Role-based access control must be enforced at controller level.
- Ownership validation must be enforced at service layer.
- Unauthorized access must return HTTP 403.

---

### 4.3 Input Validation

- All input DTOs must be validated using server-side validation.
- SQL injection must be prevented via JPA prepared statements.
- Invalid enum values must be rejected.

---

### 4.4 File Handling Security

- Resume URLs must be validated.
- Backend must not store file binaries.
- Backend must reject suspicious URL domains.

---

## 5. Reliability Requirements

### 5.1 Availability

- Target uptime: ≥ 95% (aligned with free-tier hosting constraints).
- System must recover automatically after server restart.

---

### 5.2 Data Integrity

- Database must enforce foreign key constraints.
- Unique constraint must prevent duplicate applications.
- Status history must accurately reflect transitions.

---

### 5.3 Error Handling

- The system must return structured error responses.
- Internal exceptions must not expose stack traces in production.

---

## 6. Maintainability Requirements

### 6.1 Code Structure

- Follow layered architecture:
  - Controller
  - Service
  - Repository
- Business logic must not reside in controllers.
- Entities must not be directly exposed to API responses.

---

### 6.2 Documentation

- APIs must be documented using OpenAPI (Swagger).
- Code must follow consistent naming conventions.
- Database migrations must be version-controlled (Flyway).

---

### 6.3 Modularity

- Authentication logic must be modular.
- Business rules must be encapsulated in services.
- DTOs must be separated from entity models.

---

## 7. Usability Requirements

### 7.1 Frontend

- UI must clearly distinguish Candidate and Recruiter dashboards.
- Forms must provide clear validation messages.
- Pagination must be intuitive.

---

### 7.2 Accessibility

- UI must use readable font sizes.
- Buttons and actions must be clearly labeled.
- Lighthouse Accessibility score must be ≥ 90.
- Semantic HTML usage required to support basic screen readers.

---

## 8. Compatibility Requirements

- Backend must run on Java 17+.
- Database must be PostgreSQL-compatible.
- Frontend must support latest versions of Chrome and Firefox.
- API must follow REST standards.

---

## 9. Portability Requirements

- Application must be containerizable (optional but supported).
- Environment configuration must use environment variables.
- No hard-coded credentials allowed.

---

## 10. Logging & Monitoring Requirements

- System must log:
  - Authentication attempts
  - Internship creation
  - Application submission
  - Status updates
- Logs must exclude sensitive information.
- Logging level must be configurable (DEV vs PROD).

---

## 11. Backup & Recovery Requirements

- Database provider (Neon/PostgreSQL) must support backups.
- Manual backup capability must be documented.
- System must tolerate server restarts without data loss.

---

## 12. Compliance Requirements

- System must protect user data.
- Passwords must not be reversible.
- User data must not be exposed to unauthorized roles.

Formal regulatory compliance (e.g., GDPR, SOC2) is not required for MVP but best practices must be followed.

## 13. Constraints & Assumptions

Technical constraints and product assumptions are maintained in the `1. Product Requirments Document.md`. All NFRs are evaluated within those established boundaries.

## 14. Acceptance Criteria for NFR

The system meets non-functional requirements when:

- Performance metrics fall within defined thresholds.
- Duplicate applications are impossible at DB level.
- Unauthorized access is prevented.
- Server restarts do not corrupt data.
- Error responses are standardized.

---

## 15. Summary

This document defines the quality standards for the Intern Job Tracking Application.

The system must be:

- Secure
- Performant under small-scale load
- Reliable
- Maintainable
- Cleanly structured

These non-functional requirements ensure the system is professionally engineered despite its limited scope.

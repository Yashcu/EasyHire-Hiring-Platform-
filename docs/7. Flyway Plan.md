# Migration & Versioning Strategy (Flyway Plan)

## Intern Job Tracking Application

Version: 1.0  
Status: Final  

---

## 1. Introduction

### 1.1 Purpose

This document defines the database migration and versioning strategy for the Intern Job Tracking Application using **Flyway**.

The goal is to:

- Maintain schema consistency across environments
- Prevent database drift
- Ensure safe incremental schema evolution
- Support reproducible deployments

---

## 2. Migration Tool

Tool: **Flyway**

Integrated via:

- Spring Boot Flyway starter
- Automatic execution on application startup

Flyway manages:

- Schema version tracking
- Migration execution order
- Roll-forward updates

---

## 3. Migration Philosophy

### 3.1 Core Principles

1. Never modify an applied migration.
2. Always create a new migration for schema changes.
3. Migrations are immutable once deployed.
4. All schema changes must go through Flyway.
5. Manual DB changes in production are strictly prohibited.

---

## 4. Migration File Structure

Directory:

```text
src/main/resources/db/migration/
```

Naming Convention:

```text
V<Version>__<Description>.sql
```

Examples:

```text
V1__init_users_table.sql
V2__create_candidate_profiles.sql
V3__create_internships.sql
V4__create_applications.sql
V5__create_application_status_history.sql
```

---

## 5. Initial Schema Migration Plan

### 5.1 V1__create_users_table.sql

- Create users table
- Add unique constraint on email
- Add index on email

---

### 5.2 V2__create_candidate_profiles.sql

- Create candidate_profiles table
- Add FK to users
- ON DELETE CASCADE

---

### 5.3 V3__create_internships.sql

- Create internships table
- Add FK to users (recruiter_id)
- Add index on recruiter_id
- Add index on status

---

### 5.4 V4__create_applications.sql

- Create applications table
- Add FK to users (candidate_id)
- Add FK to internships
- Add UNIQUE(candidate_id, internship_id)
- Add indexes on candidate_id and internship_id

---

### 5.5 V5__create_application_status_history.sql

- Create application_status_history table
- Add FK to applications
- Add FK to users (changed_by)
- Add index on application_id

---

## 6. Flyway Schema History Table

Flyway automatically maintains:

```text
flyway_schema_history
```

This table tracks:

- Version
- Description
- Script name
- Execution timestamp
- Execution success status

This ensures:

- No duplicate migrations
- Controlled migration order
- Safe environment upgrades

---

## 7. Environment Strategy

### 7.1 Development

- Flyway runs automatically on startup.
- clean-on-validation-error allowed (optional for dev only).
- Frequent small migrations permitted.

---

### 7.2 Production

- Flyway runs automatically during deployment.
- clean disabled.
- Manual DB edits forbidden.
- Backups recommended before major releases.

---

## 8. Schema Evolution Strategy

### 8.1 Adding a Column

Create new migration:

```text
V6__add_phone_number_to_candidate.sql
```

Never modify V1–V5.

---

### 8.2 Modifying a Column

Example: increase VARCHAR length

Create migration:

```text
V7__alter_email_length.sql
```

Use ALTER TABLE statements.

---

### 8.3 Removing a Column

Create new migration:

```text
V8__drop_unused_column.sql
```

Ensure:

- No dependent code remains.
- Deployment synchronized with backend update.

---

## 9. Data Migration Strategy

If schema change requires data transformation:

- Write SQL transformation inside migration.
- Ensure transaction safety.
- Test migration locally before production.

Example:

```sql
UPDATE internships
SET status = 'OPEN'
WHERE status IS NULL;
```

---

## 10. Rollback Strategy

Flyway Community Edition does NOT support automatic rollback.

Strategy:

1. Backup database before major deployment.
2. If migration fails:
   - Fix issue
   - Create new corrective migration
3. Never manually delete migration history entries.

Rollback is handled by:

- Restoring backup

OR

- Applying forward corrective migration.

---

## 11. Versioning Policy

Versioning format:

```text
V<Version>__<Description>.sql
```

The system uses **sequential numeric versioning**:

Example:

```text
V1__initial_schema.sql
V2__add_profile_fields.sql
V3__fix_index_issue.sql
```

This ensures simplicity and guarantees execution order without complex dot-notation parsing.

---

## 12. Best Practices

- Keep migrations small and atomic.
- Avoid mixing multiple logical changes in one file.
- Test migrations against fresh DB and existing DB.
- Keep schema changes backward-compatible where possible.
- Commit migration files with code changes.

---

## 13. Testing Migrations

For each new migration:

1. Start with clean DB.
2. Run full migration chain.
3. Verify schema integrity.
4. Run application.
5. Validate data operations.

---

## 14. Failure Handling

If migration fails during deployment:

- Application startup will fail.
- Error logged in console.
- Fix migration script.
- Create corrective version.
- Redeploy.

Never:

- Edit already applied migration.
- Delete flyway_schema_history entries.

---

## 15. Database Reset Policy (Development Only)

For development environment:

Allowed:

- Drop schema
- Re-run migrations from scratch

Not allowed in production.

---

## 16. Future Schema Extensions

Planned future additions may include:

- Admin table
- Email notification logs
- Company entity
- Audit logs

Each must follow:

- New migration file
- Version increment
- Proper foreign key definition

---

## 17. Summary

This Flyway migration strategy ensures:

- Controlled schema evolution
- No database drift
- Environment consistency
- Safe production deployment
- Clean version tracking

The system maintains professional database governance standards while remaining lightweight and manageable for small-scale deployment.

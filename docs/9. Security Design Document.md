# Security Design Document

## Intern Job Tracking Application

Version: 1.0  
Status: Final  

---

## 1. Introduction

### 1.1 Purpose

This document defines the security architecture and controls implemented in the Intern Job Tracking Application.

It covers:

- Authentication mechanisms
- Authorization strategy
- Data protection
- Input validation
- Common vulnerability mitigation
- Secure configuration practices

The goal is to ensure protection against common web application threats while maintaining simplicity appropriate for small-scale deployment.

---

## 2. Security Objectives

The system must:

1. Protect user credentials.
2. Prevent unauthorized access to resources.
3. Prevent cross-user data exposure (IDOR).
4. Prevent duplicate application manipulation.
5. Protect against common OWASP Top 10 vulnerabilities.
6. Avoid sensitive data leakage.

---

## 3. Authentication Design

### 3.1 Authentication Mechanism

Authentication is implemented using:

- Spring Security
- JWT (JSON Web Tokens)
- BCrypt password hashing

---

### 3.2 Password Security

- Passwords are hashed using BCrypt.
- Plaintext passwords are never stored.
- Password hashes are not logged.
- Password comparison uses secure hashing verification.

---

### 3.3 JWT Token Design

JWT payload includes:

- user_id
- role
- expiration timestamp

JWT characteristics:

- Signed using HMAC (server secret).
- Expiration enforced (e.g., 24 hours).
- Stateless authentication (no server session storage).

---

### 3.4 Token Validation

On each protected request:

1. Authorization header is validated.
2. JWT signature verified.
3. Expiration checked.
4. User identity extracted.
5. Role attached to security context.

Invalid tokens result in:

- 401 Unauthorized

---

## 4. Authorization Strategy

### 4.1 Role-Based Access Control (RBAC)

Roles:

- CANDIDATE
- RECRUITER

Access is restricted at controller level using role checks.

---

### 4.2 Ownership-Based Authorization

Role validation alone is insufficient.

Ownership validation is enforced in service layer:

- Candidate can only access own applications.
- Recruiter can only access internships they created.
- Recruiter can only update applications for their internships.

Unauthorized access results in:

- 403 Forbidden

---

## 5. Data Protection

### 5.1 Sensitive Data Handling

Sensitive Data:

- Password hashes
- JWT secret
- Database credentials

Protection Measures:

- Environment variables for secrets.
- No hard-coded credentials.
- No sensitive data returned in API responses.

---

### 5.2 Resume Handling

- Backend stores only resume URL string.
- Files are stored externally (Cloudinary).
- Resume URL must be validated.
- Backend must not accept arbitrary file uploads.

---

## 6. Input Validation

### 6.1 DTO Validation

All incoming requests are validated using:

- Bean Validation annotations
- Required field checks
- Enum validation
- UUID validation

Invalid input results in:

- 400 Bad Request

---

### 6.2 SQL Injection Prevention

- JPA uses prepared statements.
- No dynamic raw SQL construction.
- No direct string concatenation in queries.

---

### 6.3 XSS Prevention

- Backend returns JSON only.
- No HTML rendering.
- Frontend responsible for escaping user-generated content.

---

## 7. Protection Against OWASP Top 10

| Vulnerability | Mitigation |
| ------------- | ---------- |
| Broken Authentication | BCrypt + JWT expiration |
| Broken Access Control | Role + ownership checks |
| Injection | JPA prepared statements |
| Security Misconfiguration | Environment variables |
| Sensitive Data Exposure | Hashing + HTTPS |
| IDOR | Ownership validation |
| CSRF | Stateless JWT + no cookies |
| Logging of sensitive data | Filtered logging |

---

## 8. IDOR Prevention

Critical Rule:

Never trust user-provided IDs for authorization.

Example:

Instead of:

`GET /applications?candidateId=xyz`

System uses:

- candidate_id extracted from JWT.

This prevents cross-user data access.

---

## 9. Duplicate Application Protection

Enforced at:

Database level:

- `UNIQUE(candidate_id, internship_id)`

Application level:

- Pre-check before insert

Even under concurrent requests, DB constraint prevents duplicates.

---

## 10. Exception & Error Handling Security

- Stack traces are not exposed in production.
- Generic error messages returned.
- Validation errors are structured and safe.

---

## 11. Configuration Security

### 11.1 Secret Management

Stored in:

- Environment variables
- Production environment configuration

Never stored in:

- Git repository
- Source code
- Public logs

---

### 11.2 HTTPS Enforcement

- All production traffic must use HTTPS.
- JWT tokens transmitted only over secure channel.

---

## 12. Logging Security

The system logs:

- Login attempts
- Application submissions
- Status updates

The system does NOT log:

- Passwords
- JWT tokens
- Full sensitive payloads

---

## 13. Account Management

### 13.1 User Activation & Email Verification

- Users are created in an inactive or unverified state.
- An email containing a verification token/link is immediately dispatched.
- Accounts are marked active only upon successful verification to prevent spam and fraudulent applications.
- Users cannot log in or apply to jobs until their email is verified.

### 13.2 Password Reset (Future Scope)

Not included in MVP.
Would require:

- Secure token generation
- Expiring reset tokens
- Email integration

---

## 14. Denial of Service (DoS) & Abuse Mitigation

Given it is an open internet application:

- **Basic Rate Limiting (IP-Based):** Implemented via a lightweight filter (e.g., Bucket4j or Redis-based) to limit requests on `/auth/*` and `/internships/*/apply` endpoints.
- Database constraints protect duplicate spam.
- Free-tier infrastructure inherently limits traffic, but application-level rate limiting prevents exhausting those limits artificially.

Future enhancement:

- Distributed Redis-based rate limiting if deployed on multiple nodes.

---

## 15. Security Testing Strategy

Before release:

- Test invalid JWT
- Test expired JWT
- Test role misuse
- Test cross-user access attempt
- Test duplicate application attempt
- Test invalid resume URL submission

---

## 16. Residual Risks

| Risk | Mitigation |
| ---- | ---------- |
| JWT secret leak | Rotate secret |
| Free-tier exposure | Acceptable risk |
| Resume URL spoofing | Validate domain |
| Brute force login | Basic rate limiting implemented; advanced distributed protection not implemented. |

---

## 17. Security Limitations

- No refresh token rotation.
- No MFA.
- No intrusion detection.
- No enterprise compliance certifications.

Acceptable due to:

- Small-scale deployment.
- Intern-focused usage.
- Non-sensitive data domain.

---

## 18. Summary

The system implements:

- Secure password storage
- Stateless JWT authentication
- Role-based access control
- Ownership validation
- Input validation
- Database-level integrity enforcement

It protects against common web vulnerabilities while maintaining architectural simplicity appropriate for its scope.

This document defines the formal security architecture of the Intern Job Tracking Application.

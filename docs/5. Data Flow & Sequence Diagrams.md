# Data Flow & Sequence Diagrams

**Intern Job Tracking Application**  
Version: 1.0  
Status: Final  

---

## 1. Introduction

This document describes how data moves through the system and provides sequence-level interaction flows for core operations.

The purpose is to clearly illustrate:

- Request lifecycle
- Authentication flow
- Application submission flow
- Status update flow
- Ownership validation logic

The system follows a monolithic layered architecture:

```text
Client (React)
│
▼
Spring Boot API
│
▼
PostgreSQL Database
```

---

## 2. High-Level Data Flow Diagram (DFD)

### 2.1 Context-Level DFD (Level 0)

```text
+-------------------+             +-----------------------------+
|     Candidate     |             |          Recruiter          |
+-------------------+             +-----------------------------+
          │                                     │
          │      HTTP Requests (REST API)       │
          └───────────────┬─────────────────────┘
                          ▼
                +---------------------+
                | Spring Boot Backend |
                +---------------------+
                          │
                          │ SQL Queries
                          ▼
                +---------------------+
                |    PostgreSQL DB    |
                +---------------------+
```

---

### 2.2 Logical Data Flow (Level 1)

#### Authentication Flow

```text
User → POST /auth/login → Controller
Controller → AuthService
AuthService → UserRepository → DB
DB → AuthService
AuthService → Generate JWT
Controller → Return JWT → User
```

---

#### Internship Application Flow

```text
Candidate → POST /internships/{id}/apply
→ ApplicationController
→ ApplicationService
→ InternshipRepository (validate OPEN)
→ ApplicationRepository (check duplicate)
→ DB (insert application)
→ DB (insert history record)
→ Return success
```

---

#### Recruiter Status Update Flow

```text
Recruiter → PATCH /applications/{id}/status
→ ApplicationController
→ ApplicationService
→ ApplicationRepository (fetch)
→ Ownership Validation
→ DB (update status)
→ DB (insert history)
→ Return updated status
```

## 3. Detailed Sequence Diagrams

---

### 3.1 User Registration Sequence

```text
       User
         │
         │ POST /auth/register
         ▼
  AuthController
         │
         │ validate DTO
         ▼
   AuthService
         │
         │ hash password (BCrypt)
         │ save user
         ▼
 UserRepository
         │
         ▼
    PostgreSQL
         │
         ▼
   AuthService
         │
         │ create profile record
         ▼
ProfileRepository
         │
         ▼
    PostgreSQL
         │
         ▼
  AuthController
         │
         │ return success
         ▼
       User
```

Key Validation Points:

- Unique email check
- Role validation
- Password hashing

---

### 3.2 Login & JWT Issuance Sequence

```text
       User
         │
         │ POST /auth/login
         ▼
  AuthController
         │
         ▼
   AuthService
         │
         │ find user by email
         ▼
 UserRepository → PostgreSQL
         │
         │ compare password (BCrypt)
         │ generate JWT
         ▼
  AuthController
         │
         │ return JWT
         ▼
       User
```

JWT contains:

- user_id
- role
- expiration timestamp

---

### 3.3 Browse Internships Sequence

```text
Candidate (or Public)
         │
         │ GET /internships?page=0&size=10
         ▼
InternshipController
         │
         ▼
 InternshipService
         │
         ▼
InternshipRepository
         │
         ▼
    PostgreSQL
         │
         ▼
 InternshipService
         │
         ▼
InternshipController
         │
         ▼
       Client
```

Business Rules:

- Only internships with status = OPEN are returned
- Pagination applied at DB level

---

### 3.4 Apply to Internship Sequence

```text
    Candidate
         │
         │ POST /internships/{id}/apply
         ▼
ApplicationController
         │
         │ Validate JWT & role
         ▼
 ApplicationService
         │
         │ Fetch internship
         │ Verify internship is OPEN
         │ Check duplicate (candidate_id, internship_id)
         ▼
ApplicationRepository
         │
         ▼
    PostgreSQL (UNIQUE constraint enforcement)
         │
         ▼
 ApplicationService
         │
         │ Insert application_status_history record
         ▼
 HistoryRepository
         │
         ▼
    PostgreSQL
         │
         ▼
ApplicationController
         │
         ▼
    Candidate
```

Failure Cases:

- Internship not OPEN → 400
- Duplicate application → 409
- Unauthorized role → 403

---

### 3.5 View Candidate Applications Sequence

```text
    Candidate
         │
         │ GET /candidate/applications
         ▼
ApplicationController
         │
         ▼
 ApplicationService
         │
         │ Query by candidate_id (from JWT)
         ▼
ApplicationRepository
         │
         ▼
    PostgreSQL
         │
         ▼
 ApplicationService
         │
         ▼
ApplicationController
         │
         ▼
    Candidate
```

Critical Check:

- Must use candidate_id from JWT
- Not from request parameter (prevents IDOR)

---

### 3.6 Recruiter View Applications Sequence

```text
    Recruiter
         │
         │ GET /recruiter/internships/{id}/applications
         ▼
ApplicationController
         │
         ▼
 ApplicationService
         │
         │ Verify internship owner
         │ Query applications by internship_id
         ▼
ApplicationRepository
         │
         ▼
    PostgreSQL
         │
         ▼
 ApplicationService
         │
         ▼
ApplicationController
         │
         ▼
    Recruiter
```

Ownership Validation:

- Internship.recruiter_id must match JWT user_id

---

### 3.7 Update Application Status Sequence

```text
    Recruiter
         │
         │ PATCH /applications/{id}/status
         ▼
ApplicationController
         │
         ▼
 ApplicationService (Transactional)
         │
         │ Fetch application
         │ Validate recruiter ownership
         │ Validate status transition
         │ Update application status
         ▼
ApplicationRepository → PostgreSQL
         │
         │ Insert history record
         ▼
 HistoryRepository → PostgreSQL
         │
         ▼
ApplicationController
         │
         ▼
    Recruiter
```

Atomicity:

- Update + history insert must occur in single transaction.

## 4. Data Ownership Flow

| Resource | Owner | Validation Mechanism |
| -------- | ----- | -------------------- |
| Application | Candidate | candidate_id from JWT |
| Internship | Recruiter | recruiter_id from JWT |
| Status Update | Recruiter | Ownership of internship |

## 5. Error Handling Flow

```text
Request → Controller
→ Service
→ Exception Thrown
→ Global Exception Handler
→ Structured Error Response
```

Error Response Format:

- HTTP Status
- Message
- Timestamp
- Request Path

## 6. Security Control Points

| Layer | Security Enforcement |
| ----- | -------------------- |
| Filter Chain | JWT validation |
| Controller | Role check |
| Service | Ownership validation |
| Database | Foreign keys + Unique constraint |

---

## 7. Summary

The system follows a clear, layered request lifecycle:

1. Request enters via Controller.
2. Authentication enforced at filter level.
3. Business rules enforced in Service layer.
4. Data integrity enforced at Database level.
5. Structured response returned to client.

These data flows and sequence diagrams ensure:

- No cross-user data leakage.
- Clean transaction handling.
- Proper authorization enforcement.
- Consistent lifecycle for all major operations.

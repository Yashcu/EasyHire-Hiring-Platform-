# Development Roadmap

## Intern Job Tracking Application

Version: 1.0
Status: Final

---

## 1. Introduction

### 1.1 Purpose

This document defines the structured development roadmap for the Intern Job Tracking Application.

It outlines:

- Development phases
- Milestones
- Deliverables
- Dependencies
- Validation checkpoints

The roadmap is optimized for a single-developer workflow and small-scale deployment.

---

## 2. Development Strategy

Guiding Principles:

1. Backend-first development.
2. Complete one vertical feature before moving to next.
3. Maintain working state after each milestone.
4. Avoid premature optimization.
5. Deploy early, iterate safely.

---

## 3. Phase Overview

| Phase | Focus | Outcome |
| ----- | ----- | ------- |
| Phase 1 | Project Setup | Working backend skeleton |
| Phase 2 | Authentication | Secure login & registration |
| Phase 3 | Internship Module | Recruiter CRUD complete |
| Phase 4 | Application Module | Candidate apply workflow |
| Phase 5 | Status & History | Recruiter workflow complete |
| Phase 6 | Frontend | Full UI integration |
| Phase 7 | Deployment | Production-ready system |
| Phase 8 | Testing & Refinement | Stability & polish |

---

## 4. Phase 1 – Project Setup

### 4.1 Objectives

- Initialize Spring Boot project.
- Configure PostgreSQL connection.
- Integrate Flyway.
- Establish layered architecture.
- Configure Swagger.

### 4.2 Deliverables

- Project compiles and runs.
- Database connection verified.
- First migration applied.
- Swagger accessible.

### 4.3 Completion Criteria

- `/swagger-ui.html` loads.
- Flyway migration runs successfully.

Estimated Duration: 1–2 days

---

## 5. Phase 2 – Authentication Module

### 5.1 Objectives

- Implement User entity.
- Implement registration endpoint.
- Implement login endpoint.
- Integrate BCrypt hashing.
- Implement JWT generation.
- Configure Spring Security.

### 5.2 Deliverables

- User can register.
- User can log in.
- Protected endpoint requires JWT.
- Role-based restriction working.

### 5.3 Validation Tests

- Invalid login fails.
- Expired token rejected.
- Role mismatch denied.

Estimated Duration: 2–3 days

---

## 6. Phase 3 – Internship Module

### 6.1 Objectives

- Create Internship entity.
- Implement recruiter-only creation.
- Implement update endpoint.
- Implement public listing endpoint.
- Implement pagination.

### 6.2 Deliverables

- Recruiter can create internship.
- Public users can view internships.
- Only OPEN internships visible.

### 6.3 Validation Tests

- Candidate cannot create internship.
- Non-owner recruiter cannot update.

Estimated Duration: 2–3 days

---

## 7. Phase 4 – Application Module

### 7.1 Objectives

- Create Application entity.
- Implement apply endpoint.
- Add UNIQUE constraint.
- Store resume URL.
- Implement candidate applications retrieval.

### 7.2 Deliverables

- Candidate can apply.
- Duplicate application rejected.
- Candidate can view own applications.

### 7.3 Validation Tests

- Applying twice returns 409.
- Candidate cannot access another candidate’s data.

Estimated Duration: 2–3 days

---

## 8. Phase 5 – Status & History Module

### 8.1 Objectives

- Implement application_status_history.
- Implement status update endpoint.
- Enforce valid status transitions.
- Enforce recruiter ownership validation.

### 8.2 Deliverables

- Recruiter can update application status.
- History entries recorded.
- History retrieval endpoint functional.

### 8.3 Validation Tests

- Invalid status transition rejected.
- Status update atomic with history insert.

Estimated Duration: 2–3 days

---

## 9. Phase 6 – Frontend Development

### 9.1 Objectives

- Setup Vite + React + Tailwind.
- Implement AuthContext.
- Implement protected routes.
- Build candidate dashboard.
- Build recruiter dashboard.
- Integrate with backend API.

### 9.2 Deliverables

- End-to-end flow working:
  - Register
  - Login
  - Create internship
  - Apply
  - Update status

### 9.3 Validation Tests

- Role-based UI restriction works.
- Token expiration handled properly.

Estimated Duration: 4–6 days

---

## 10. Phase 7 – Deployment

### 10.1 Objectives

- Deploy backend to Render.
- Deploy frontend to Vercel.
- Configure environment variables.
- Verify Flyway migration in production.
- Conduct smoke testing.

### 10.2 Deliverables

- Public URL accessible.
- Full workflow tested in production.

Estimated Duration: 1–2 days

## 11. Phase 8 – Testing & Refinement

### 11.1 Objectives

- Optimize minor performance issues.
- Clean logs and remove debug code.

### 11.2 Deliverables

- Updated README.

## 14. Risk Mitigation During Development

| Risk | Mitigation |
| ---- | ---------- |
| Scope creep | Strict adherence to PRD |
| Overengineering | Avoid unnecessary infrastructure |
| Security misconfiguration | Test JWT and role restrictions |
| Schema errors | Use Flyway consistently |

---

## 15. Definition of Done (DoD)

The system is considered complete when:

- All functional requirements are implemented.
- All protected endpoints enforce role and ownership.
- Duplicate applications are impossible.
- Application history is accurate.
- Frontend fully integrates with backend.
- Production deployment is stable.

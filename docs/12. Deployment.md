Version: 1.0

## 1. Introduction

### 1.1 Purpose

This document describes the deployment process for the Intern Job Tracking Application.

It covers:

- Environment setup
- Backend deployment
- Database configuration
- Frontend deployment
- Environment variables
- Production verification steps

The system is designed for deployment using free-tier cloud services.

## 4. Backend Deployment (Spring Boot)

### 4.1 Prerequisites

- Java 17+
- Maven or Gradle
- PostgreSQL database (Neon)
- Render account

---

### 4.2 Build Configuration

Ensure production profile exists:

`application-prod.yml`

Disable:

- Debug logs
- Stack trace exposure

---

### 4.3 Required Environment Variables

Set in Render dashboard:

```bash
SPRING_DATASOURCE_URL=jdbc:postgresql://<host>:5432/<db>
SPRING_DATASOURCE_USERNAME=<username>
SPRING_DATASOURCE_PASSWORD=<password>

JWT_SECRET=<secure_random_string>

SPRING_PROFILES_ACTIVE=prod

CORS_ALLOWED_ORIGINS=https://your-frontend-domain.vercel.app
RATE_LIMIT_CAPACITY=100
RATE_LIMIT_REFILL_TOKENS=10
```

Optional:

```bash
CLOUDINARY_CLOUD_NAME=<cloud>
CLOUDINARY_API_KEY=<key>
CLOUDINARY_API_SECRET=<secret>
```

---

### 4.4 Deploying to Render

#### Step 1: Connect Git Repository

- Connect GitHub repository to Render.
- Select backend repository.

#### Step 2: Configure Service

- Environment: Java
- Build Command:

```bash
./mvnw clean package -DskipTests
```

- Start Command:

```bash
java -jar target/*.jar
```

#### Step 3: Add Environment Variables

Add all required secrets securely.

#### Step 4: Deploy

- Render automatically builds and deploys.

---

### 4.5 Flyway Migration Execution

On application startup:

- Flyway automatically runs migrations.
- `flyway_schema_history` table is created.
- Schema version validated.

If migration fails:

- Deployment fails.
- Fix migration and redeploy.

## 9. Backup & Recovery

### 9.1 Database Backups

Neon provides:

- Automated backups (depending on tier).

Manual backup:

- Export via `pg_dump` if needed.

---

### 9.2 Rollback Strategy

If deployment fails:

1. Identify migration error.
2. Fix code or create new migration.
3. Redeploy.

If schema corruption occurs:

- Restore database from backup.

---

## 10. Scaling Considerations (Future)

If user base grows:

- Upgrade Neon DB plan.
- Upgrade Render instance.
- Introduce Redis for caching.
- Add rate limiting.

Not required for current scope.

---

## 11. Security in Deployment

- Never commit secrets to Git.
- Rotate JWT secret if exposed.
- Use secure HTTPS endpoints only.
- Restrict database access via provider controls.

---

## 12. Monitoring & Logs

Render provides:

- Application logs.
- Deployment logs.

Recommended:

- Monitor error logs regularly.
- Check DB connection errors.
- Monitor memory usage.

---

## 13. Common Deployment Issues

| Issue | Cause | Fix |
| ----- | ----- | --- |
| 401 errors | JWT secret mismatch | Verify secret |
| DB connection error | Incorrect URL | Check connection string |
| CORS error | Backend CORS misconfigured | Configure allowed origins |
| Flyway failure | Broken migration | Fix and redeploy |

---

## 14. Deployment Flow Summary

1. Push backend code to Git.
2. Render builds and deploys.
3. Flyway runs migrations.
4. Deploy frontend to Vercel.
5. Configure API base URL.
6. Verify end-to-end functionality.

---

## 15. Conclusion

The deployment strategy:

- Uses free-tier infrastructure.
- Maintains secure configuration.
- Automates schema migrations.
- Supports clean environment separation.
- Enables quick iteration.

This guide ensures consistent, repeatable, and secure deployment of the Intern Job Tracking Application.

# Functional Requirements Specification (FRS)

**Intern Job Tracking Application**  
Version: 1.0  
Status: Final  

---

## 1. Introduction

### 1.1 Purpose

This document defines the detailed functional behavior of the Intern Job Tracking Application. It describes system actions, business rules, validations, and user interactions required to fulfill the Product Requirements Document (PRD).

This document focuses strictly on **what the system must do**, not how it is implemented.

---

## 2. System Actors

| Actor | Description |
| ----- | ----------- |
| Candidate | User applying for internships |
| Recruiter | User posting internships and reviewing applications |
| Admin (Optional) | System moderator (future scope) |
| System | Backend processing unit |

---

## 3. Authentication & Authorization Requirements

### 3.1 User Registration

#### FR-1 Register User

The system shall allow a new user to register with:

- Email (unique)
- Password
- Role (CANDIDATE or RECRUITER)

#### FR-1 Business Rules

- Email must be unique.
- Password must be securely stored (hashed).
- User role must be explicitly selected during registration.
- A corresponding profile record must be created.

---

### 3.2 User Login

#### FR-2 Login

The system shall authenticate a registered user and issue a JWT token upon valid credentials.

#### FR-2 Business Rules

- Invalid credentials must return an authentication error.
- Token must include role information.
- Token must be required for all protected endpoints.

---

### 3.3 Authorization

#### FR-3 Role Enforcement

The system shall restrict endpoints based on user role.

| Endpoint | Allowed Role |
| -------- | ------------ |
| Create Internship | RECRUITER |
| Apply to Internship | CANDIDATE |
| Update Application Status | RECRUITER |
| View Own Applications | CANDIDATE |
| View Internship Applications | RECRUITER |

#### FR-4 Ownership Validation

The system shall verify ownership when:

- Candidate accesses application details.
- Recruiter accesses internship-specific applications.
- Recruiter updates application status.

Unauthorized access must return 403 Forbidden.

---

## 4. Candidate Functional Requirements

### 4.1 Browse Internships

#### FR-5 View Internship Listings

The system shall allow public viewing of internship listings with:

- Title
- Description
- Location
- Stipend Range
- Type
- Status (OPEN only)

#### FR-6 Pagination

The system shall support paginated results.

---

### 4.2 Apply for Internship

#### FR-7 Submit Application

The system shall allow a candidate to apply to an OPEN internship.

Required Inputs:

- Internship ID
- Resume URL (Cloudinary)

#### FR-7 Business Rules

- Candidate must be authenticated.
- Internship must be OPEN.
- Resume URL must not be empty.
- Candidate may apply only once per internship.

#### FR-8 Duplicate Prevention

The system shall enforce uniqueness of:
(candidate_id, internship_id)

Duplicate applications must return an error.

#### FR-9 Application Creation

Upon successful submission:

- Application record must be created.
- Status must be set to APPLIED.
- applied_at timestamp must be stored.

---

### 4.3 View Own Applications

#### FR-10 Retrieve Applications

The system shall allow candidates to view only their own applications.

Response must include:

- Internship details
- Current status
- Applied timestamp

---

### 4.4 View Application History

#### FR-11 Retrieve Status History

The system shall allow candidates to view the status history of their own applications.

Each history record must include:

- Previous status
- New status
- Changed by (recruiter ID)
- Timestamp

---

## 5. Recruiter Functional Requirements

### 5.1 Create Internship

#### FR-12 Create Internship Requirement

The system shall allow recruiters to create internship postings.

Required Fields:

- Title
- Description
- Location
- Stipend range
- Type (REMOTE/ONSITE/HYBRID)

#### FR-12 Business Rules

- Internship status defaults to DRAFT or OPEN.
- Recruiter ID must be stored as owner.

---

### 5.2 Update Internship

#### FR-13 Update Internship Requirement

The system shall allow recruiters to update only internships they own.

Unauthorized update must return 403 Forbidden.

---

### 5.3 Change Internship Status

#### FR-14 Update Internship Status

The system shall allow recruiters to change internship status:

- DRAFT → OPEN
- OPEN → CLOSED
- CLOSED → OPEN (Reopening allows new applications)
- DRAFT → CLOSED (Cancellation before opening)

Closed internships must not accept applications.

---

### 5.4 View Applications for Internship

#### FR-15 Retrieve Internship Applications

The system shall allow recruiters to view applications for internships they own.

Response must include:

- Candidate profile information
- Resume URL
- Application status
- Applied timestamp

Pagination must be supported.

---

### 5.5 Update Application Status

#### FR-16 Update Status

The system shall allow recruiters to update application status to:

- IN_REVIEW
- SHORTLISTED
- REJECTED
- OFFERED

#### FR-16 Business Rules

- Only recruiter owning internship may update status.
- Status must follow this precise transition matrix:
  - APPLIED → IN_REVIEW, REJECTED
  - IN_REVIEW → SHORTLISTED, REJECTED
  - SHORTLISTED → OFFERED, REJECTED
  - OFFERED, REJECTED → No further transitions (Terminal states)
- Status update must create a history record.

---

## 6. Application Status History Requirements

### 6.1 Status History Logic

#### FR-17 Status History Record

Each status change must create a history entry containing:

- Application ID
- Previous status
- New status
- Changed by
- Timestamp

Initial application creation may optionally log:

- previous_status = NULL
- new_status = APPLIED

---

## 7. Data Validation Requirements

### 7.1 Field Validation Requirements

#### FR-18 Field Validation

The system shall validate:

- Non-empty required fields
- Valid enum values
- Valid UUID format
- Valid resume URL format

Invalid input must return structured error response.

---

## 8. Error Handling Requirements

### 8.1 Error Returns

#### FR-19 Standard Error Format

The system shall return structured error responses including:

- HTTP status
- Error message
- Timestamp
- Path

#### FR-20 Forbidden Access

Accessing unauthorized resources must return 403.

#### FR-21 Not Found

Requesting non-existent entity must return 404.

---

## 9. System Logging Requirements

### 9.1 Required Logs

#### FR-22 Log Critical Events

The system shall log:

- User registration
- Login attempts
- Internship creation
- Application submission
- Status changes

Sensitive data (passwords) must not be logged.

---

## 10. File Handling Requirements

### 10.1 File Uploads

#### FR-23 Resume URL Storage

The system shall store only the resume URL string.
Files shall not be stored in the backend.

#### FR-24 Resume URL Validation

The system shall validate resume URLs to ensure:

- Non-empty
- Correct format
- Valid domain (Cloudinary)

---

## 11. Administrative Requirements (Future Scope)

### 11.1 Admin Features

#### FR-25 Admin Controls (Optional)

The system may later support:

- User deactivation
- Internship moderation
- Audit review

Not required for MVP.

---

## 12. Deletion Policy

### 12.1 Soft vs Hard Deletes

#### FR-26 Deletion Behavior

The system performs **Hard Deletes** for MVP.

- Deleting a user permanently removes their profile, internships, and applications.
- Users cannot explicitly deactivate accounts (Future Scope).
- If soft deletes are implemented later, a `deleted_at` column will be used, and soft-deleted entities must be excluded from all GET responses.
- However, for the current scope, physical deletion is the standard.

---

## 13. Functional Completion Criteria

The system is functionally complete when:

- Candidates can register, log in, browse internships, and apply.
- Recruiters can create internships and manage applications.
- Duplicate applications are prevented.
- Status changes are logged.
- Ownership checks prevent unauthorized access.

---

## 14. Summary

This FRS defines all system behaviors required for the Intern Job Tracking Application.

It ensures:

- Clear separation of roles.
- Strong validation and business rules.
- Complete candidate-to-recruiter workflow.
- Structured audit tracking.
- Controlled access to sensitive resources.

This document serves as the functional contract between product and engineering.

# Product Requirements Document (PRD)

**Intern Job Tracking Application**  
Version: 1.0  
Status: Final  
Document Owner: Product  

---

## 1. Purpose

This document defines the product vision, objectives, scope, and requirements for the **Intern Job Tracking Application**. The system is designed as a lightweight, role-based web platform enabling candidates to apply for internships and recruiters to manage applications efficiently.

This document serves as the authoritative reference for product direction and feature alignment.

---

## 2. Product Overview

### 2.1 Product Vision

To build a clean, secure, and structured internship management platform that demonstrates professional backend architecture while remaining lightweight and practical for small-scale use (campus / intern-level usage).

### 2.2 Target Audience

- University students seeking internships
- Recruiters posting internship opportunities
- Small organizations managing intern hiring

### 2.3 Scale Assumptions

- Fewer than 500 active users
- Fewer than 50 active internships at any given time
- Fewer than 5,000 total lifetime applications
- Monolithic backend deployment

---

## 3. Objectives

### 3.1 Primary Objectives

1. Provide secure user authentication and role-based access.
2. Enable recruiters to create and manage internship postings.
3. Enable candidates to apply for internships with resume submission.
4. Allow recruiters to review and update application status.
5. Maintain application history integrity.

### 3.2 Secondary Objectives

1. Provide structured audit history for application status changes.
2. Maintain clean backend architecture suitable for interviews/portfolio.
3. Ensure safe data handling and protection against common vulnerabilities.

---

## 4. Success Metrics (KPIs)

| Metric | Target |
| ------ | ------ |
| Application Submission Success Rate | > 99% |
| API Error Rate | < 2% |

## 5. User Roles

### 5.1 Candidate

- Register and log in
- Create profile
- Browse internships
- Apply for internships
- View application status
- View status history

### 5.2 Recruiter

- Register and log in
- Create internships
- Update internships
- View applicants
- Change application status
- View application history

### 5.3 Admin (Optional / Future)

- Manage users
- Moderate content
- View system-level insights

---

## 6. Problem Statement

Current intern tracking processes (emails, spreadsheets) lack:

- Centralized tracking
- Status visibility
- Structured workflow
- Data consistency
- Audit trail

This system provides a structured, secure alternative.

---

## 7. Core Features

### 7.1 Authentication

- User registration (Email Verification required)
- Secure login (JWT-based)
- Password hashing
- Role-based authorization

### 7.2 Internship Management

- Recruiter can create internship
- Recruiter can update internship
- Recruiter can close internship
- Public listing with pagination

### 7.3 Application Flow

- Candidate can apply once per internship
- Resume URL required (Cloudinary)
- Unique constraint prevents duplicate applications

### 7.4 Application Management

- Recruiter views applications per internship
- Recruiter updates status
- Status change recorded in history table

### 7.5 Application History

- Tracks:
  - Previous status
  - New status
  - Changed by
  - Timestamp

---

## 8. Out of Scope (Explicitly Not Included)

- Payment processing
- Multi-company enterprise tenancy
- Advanced analytics dashboards
- AI-based candidate scoring
- Microservices architecture
- Distributed messaging systems
- Real-time WebSockets
- Multi-language support
- Advanced search engine (ElasticSearch)

---

## 9. Constraints

### 9.1 Technical Constraints

- Spring Boot monolith
- PostgreSQL database
- JWT authentication
- Cloudinary file hosting
- Free-tier hosting environment

### 9.2 Resource Constraints

- Single developer project
- Limited infrastructure budget
- Designed for learning and demonstration

---

## 10. Assumptions

1. Users may attempt to abuse the open application, making basic IP-based rate limiting essential.
2. Infrastructure free-tier limits are acceptable.
3. System is not expected to handle enterprise-level traffic.
4. Resume files are hosted externally (not stored in backend).

---

## 11. Risks

| Risk | Impact | Mitigation |
| ---- | ------ | ---------- |
| Resume URL manipulation | Medium | Validate Cloudinary domain |
| Duplicate application race | Low | DB unique constraint |
| Unauthorized data access | High | Ownership checks |
| Bot traffic and spam | High | IP-based rate limiting on auth endpoints |
| Fraudulent applications | Medium | Email verification required |
| Infrastructure downtime | Medium | Acceptable for free tier |

## 13. Acceptance Criteria (High-Level)

The product is considered complete when:

- All defined user roles can execute their flows end-to-end.
- Duplicate applications are prevented at DB level.
- Application status changes are logged correctly.
- Ownership validation prevents IDOR.
- Deployment works in production environment.

## 15. Final Positioning

This product is:

- A structured intern-focused application management system.
- Architecturally clean.
- Professionally documented.
- Designed for small-scale real usage and portfolio demonstration.

It is not positioned as an enterprise ATS platform.

## 5. Routing Architecture

### 5.1 Public Routes

- `/login`
- `/register`
- `/internships`

Accessible without authentication.

---

### 5.2 Protected Routes

Candidate:

- `/candidate/dashboard`
- `/candidate/applications`

Recruiter:

- `/recruiter/dashboard`
- `/recruiter/internships`
- `/recruiter/applications/:internshipId`

Protected routes use `ProtectedRoute` component.

---

### 5.3 Role-Based Routing

ProtectedRoute performs:

1. JWT existence check.
2. Role validation.
3. Redirect if unauthorized.

Example Logic:

- Candidate cannot access recruiter routes.
- Recruiter cannot access candidate dashboard.

## 6. Authentication Flow (Frontend)

### 6.1 Login Process

1. User submits login form.
2. API call made to `/auth/login`.
3. JWT token received.
4. Token stored in:
   - localStorage (simple implementation)
5. Axios interceptor attaches token to future requests.
6. User redirected to role-based dashboard.

---

### 6.2 Token Expiry UX & Unsaved Form State

If a token expires mid-session (e.g., user is filling out a form):

1. **Detection:** Axios interceptor catches the `401 Unauthorized` response.
2. **State Preservation:** Before redirecting, the frontend pushes the current route and any unsaved form state to `sessionStorage` or a lightweight global store.
3. **Re-authentication Modal vs Redirect:**
   - The user is presented with a "Session Expired" overlay/modal so they don't lose context, asking them to re-enter their password.
   - Upon successful re-authentication, the original request is retried seamlessly.
   - If they cancel the modal or close the tab, state is naturally purged with the session, and they return to the main login page.

---

### 6.3 Logout Process

1. Remove token from storage.
2. Clear auth context and session cache.
3. Redirect to login page.

## 10. Resume Upload Flow

1. Candidate uploads resume directly to Cloudinary.
2. Cloudinary returns secure URL.
3. URL sent to backend during apply request.
4. Backend stores URL only.

Frontend does not:

- Store file locally.
- Send binary data to backend.

## 11. Form Handling & Validation

Frontend validation includes:

- Required field checks
- Email format validation
- Password length validation
- Enum selection validation

Backend validation remains authoritative.

---

## 12. Error Handling Strategy

### 12.1 Token Expiration & API Errors

- **401 Unauthorized (Expired Token):** Handled globally. Triggers the re-authentication overlay. Any in-flight API requests (like form submissions) are temporarily queued until the token is refreshed, then retried.
- **400 Bad Request:** Field-level validation errors map to the associated form fields.
- **403 Forbidden:** Shown as an Access Denied full-page message.
- display structured error messages from backend.

---

### 12.2 Loading States

All async actions must:

- Show loading spinner
- Disable submit buttons during request

## 14. Security Considerations (Frontend)

- JWT stored in localStorage (acceptable for MVP).
  - *Alternative: HTTP-only cookie for production-grade security to mitigate XSS JWT theft.*
- No sensitive data stored in frontend state.
- No rendering of raw HTML from backend.
- Authorization logic enforced primarily in backend.

Frontend authorization is convenience, not security boundary.

---

## 15. Performance Considerations

- Pagination for large lists.
- Avoid unnecessary re-renders.
- Keep components small and modular.
- No heavy libraries.

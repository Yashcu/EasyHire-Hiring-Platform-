# Database Design Document (DDD)

## Intern Job Tracking Application

Version: 1.0  
Status: Final  

---

## 1. Introduction

### 1.1 Purpose

This document defines the relational database design for the Intern Job Tracking Application. It includes:

- Entity definitions
- Table structures
- Relationships
- Constraints
- Indexing strategy
- Data integrity rules

The system uses **PostgreSQL** as the primary database.

---

## 2. Design Principles

1. Normalized relational schema (3NF).
2. Clear separation of authentication and profile data.
3. Enforced data integrity via foreign keys and unique constraints.
4. Minimal indexing (optimized for small-scale usage).
5. UUID-based primary keys.

---

## 3. Entity Relationship Overview

```text
Users
│
├── Candidate_Profiles (1:1)
│
└── Recruiter_Profiles (1:1) [Optional Future]

Users (Recruiter)
│
└── Internships (1:N)

Internships
│
└── Applications (1:N)

Applications
│
└── Application_Status_History (1:N)
```

## 4. Table Definitions

---

### 4.1 Table: users

Purpose: Authentication and role management.

| Column | Type | Constraints | Description |
| ------ | ---- | ----------- | ----------- |
| id | UUID | PK | Unique identifier |
| email | VARCHAR(255) | UNIQUE, NOT NULL | Login email |
| password_hash | VARCHAR(255) | NOT NULL | BCrypt hash |
| role | VARCHAR(20) | NOT NULL | CANDIDATE / RECRUITER |
| is_active | BOOLEAN | DEFAULT TRUE | Account status |
| email_verified | BOOLEAN | DEFAULT FALSE | Required for login/apply |
| verification_token | VARCHAR(255) | NULLABLE | Email verification token |
| verification_expiry | TIMESTAMP | NULLABLE | Token expiration |
| created_at | TIMESTAMP | NOT NULL, DEFAULT NOW() | Creation timestamp |
| updated_at | TIMESTAMP | NOT NULL, DEFAULT NOW() | Last update timestamp |

Constraints:

- `CHECK(role IN ('CANDIDATE', 'RECRUITER', 'ADMIN'))`

Indexes:

- UNIQUE(email)

Notes:

- No profile-specific fields stored here.
- Used strictly for authentication context.

---

### 4.2 Table: candidate_profiles

Purpose: Candidate-specific profile information.

| Column | Type | Constraints | Description |
| ------ | ---- | ----------- | ----------- |
| user_id | UUID | PK, FK → users(id) | Linked user |
| first_name | VARCHAR(100) | NOT NULL | Candidate first name |
| last_name | VARCHAR(100) | NOT NULL | Candidate last name |
| university | VARCHAR(255) | NULLABLE | University name |
| default_resume_url | VARCHAR(500) | NULLABLE | Default resume link |
| updated_at | TIMESTAMP | NOT NULL, DEFAULT NOW() | Last update timestamp |

Relationship:

- One-to-one with users

Foreign Key:

- ON DELETE CASCADE

---

### 4.3 Table: internships

Purpose: Internship postings by recruiters.

| Column | Type | Constraints | Description |
| ------ | ---- | ----------- | ----------- |
| id | UUID | PK | Internship ID |
| recruiter_id | UUID | FK → users(id), NOT NULL | Owner recruiter |
| title | VARCHAR(255) | NOT NULL | Internship title |
| description | TEXT | NOT NULL | Internship description |
| location | VARCHAR(255) | NOT NULL | Location |
| stipend_min | NUMERIC(10,2) | NULLABLE | Minimum stipend |
| stipend_max | NUMERIC(10,2) | NULLABLE | Maximum stipend |
| type | VARCHAR(20) | NOT NULL | REMOTE / ONSITE / HYBRID |
| status | VARCHAR(20) | NOT NULL | DRAFT / OPEN / CLOSED |
| created_at | TIMESTAMP | NOT NULL, DEFAULT NOW() | Created timestamp |
| updated_at | TIMESTAMP | NOT NULL, DEFAULT NOW() | Last update timestamp |

Constraints:

- `CHECK (stipend_min <= stipend_max)`
- `CHECK(type IN ('REMOTE', 'ONSITE', 'HYBRID'))`
- `CHECK(status IN ('DRAFT', 'OPEN', 'CLOSED'))`

Indexes:

- INDEX(recruiter_id)
- INDEX(status)

Foreign Key:

- ON DELETE RESTRICT

Business Rule:

- Only OPEN internships accept applications.

---

### 4.4 Table: applications

Purpose: Stores candidate applications to internships.

| Column | Type | Constraints | Description |
| ------ | ---- | ----------- | ----------- |
| id | UUID | PK | Application ID |
| candidate_id | UUID | FK → users(id), NOT NULL | Candidate |
| internship_id | UUID | FK → internships(id), NOT NULL | Internship |
| status | VARCHAR(20) | NOT NULL | APPLIED / IN_REVIEW / SHORTLISTED / REJECTED / OFFERED |
| applied_resume_url | VARCHAR(500) | NOT NULL | Resume snapshot |
| applied_at | TIMESTAMP | NOT NULL, DEFAULT NOW() | Submission timestamp |
| updated_at | TIMESTAMP | NOT NULL, DEFAULT NOW() | Last update timestamp |

Constraints:

- UNIQUE(candidate_id, internship_id)
- `CHECK(status IN ('APPLIED', 'IN_REVIEW', 'SHORTLISTED', 'REJECTED', 'OFFERED'))`

Indexes:

- INDEX(candidate_id)
- INDEX(internship_id)

Foreign Keys:

- candidate_id → users(id) ON DELETE CASCADE
- internship_id → internships(id) ON DELETE CASCADE

Notes:

- Resume URL stored directly to preserve historical record.
- Unique constraint prevents duplicate applications.

---

### 4.5 Table: application_status_history

Purpose: Maintains audit trail of status changes.

| Column | Type | Constraints | Description |
| ------ | ---- | ----------- | ----------- |
| id | UUID | PK | History ID |
| application_id | UUID | FK → applications(id) | Related application |
| previous_status | VARCHAR(20) | NULLABLE | Previous status |
| new_status | VARCHAR(20) | NOT NULL | Updated status |
| changed_by | UUID | FK → users(id) ON DELETE SET NULL | Recruiter who changed status |
| changed_at | TIMESTAMP | NOT NULL | Change timestamp |

Indexes:

- INDEX(application_id)

Foreign Keys:

- application_id → applications(id) ON DELETE CASCADE
- changed_by → users(id) ON DELETE SET NULL

Business Rule:

- Created on every status change.
- May optionally record initial APPLIED state.

## 6. Relationships Summary

| Relationship | Type | Description |
| ------------ | ---- | ----------- |
| users → candidate_profiles | 1:1 | Profile extension |
| users → internships | 1:N | Recruiter posts internships |
| users → applications | 1:N | Candidate submits applications |
| internships → applications | 1:N | Internship receives applications |
| applications → application_status_history | 1:N | Status changes logged |

---

## 7. Data Integrity Rules

1. Email must be unique.
2. Candidate may apply only once per internship.
3. Internship must exist before application.
4. Application must exist before history entry.
5. Foreign keys must enforce referential integrity.

---

## 8. Indexing Strategy

Minimal indexing due to small-scale usage:

- users(email) UNIQUE
- applications(candidate_id)
- applications(internship_id)
- internships(recruiter_id)
- application_status_history(application_id)

Avoid over-indexing to reduce write overhead.

---

## 9. Transactional Considerations

- Status update and history insertion must occur in single transaction.
- Application creation and initial history record must be atomic.
- Duplicate prevention relies on DB unique constraint.

---

## 10. Deletion Policy

- Deleting a user cascades to profile and applications.
- Deleting an internship cascades to applications.
- Deleting an application cascades to status history.

For MVP, hard deletes are permitted.

---

| ----- | ------------------ |
| users | < 1,000 |
| internships | < 200 |
| applications | < 5,000 |
| application_status_history | < 20,000 |

No partitioning required.

## 13. Summary

The database design:

- Follows relational normalization.
- Enforces strong data integrity.
- Prevents duplicate applications.
- Maintains audit trail.
- Remains simple and maintainable for small-scale deployment.

This schema supports all functional requirements while avoiding unnecessary complexity.
